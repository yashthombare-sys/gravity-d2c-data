## Session: 2026-03-30 — Fix Daily Dashboard Automation

### Problem:
Co-founder had to manually sync Amazon data every day because the automation was unreliable.

### Root causes identified:
1. Amazon SP-API rate limiting — `--last4` mode made ~400 API calls (fetching order items per order)
2. `sync_mtd.py` exited with code 1 on partial failures, preventing commit of partial-but-valid data
3. MTD sync ran on fixed 8 AM cron, sometimes before Amazon sync finished
4. Both workflows shared a concurrency group, causing queuing issues

### Fixes applied:
1. `amazon_daily_mis.py` — Removed "items" from `--last4` fetch steps. Cuts API calls from ~400 to ~15. Uses OrderTotal + 36% COGS estimate instead. Monthly full sync still fetches items for accuracy.
2. `amazon_daily_mis.py` — Reduced retry delays from 20 min to 1-2 min (fast enough without items fetch)
3. `amazon_daily_sync.yml` — Reduced timeout from 90 to 20 minutes
4. `sync_mtd.py` — Removed sys.exit(1) on partial failures. Dashboard gets updated with available data.
5. `daily_sync.yml` — Added `workflow_run` trigger so MTD sync fires AFTER Amazon sync completes. Kept cron at 10 AM IST as fallback. Gave it its own concurrency group.
6. `amazon_daily_sync.yml` — Changed concurrency group from shared to `amazon-sync`
7. `keep_alive.yml` — Simplified to only check Amazon sync (MTD auto-follows via workflow_run)

### Pipeline flow (after fix):
6:00 AM IST → Amazon sync (~5-10 min) → auto-triggers MTD sync → dashboard updated
9:00 AM IST → Backup check: if Amazon didn't run, trigger it (MTD follows automatically)
10:00 AM IST → Fallback cron for MTD (in case workflow_run didn't fire)

### Key architectural notes:
- automation/daily_sync.py (SQLite pipeline) is NOT used by any workflow — dead code path
- There are 3 Google Sheets involved: D2C Daily, Amazon Daily MIS, Daily MIS (product-level)
- The dashboard.html gets data injected via JS markers (MTD_DATA_START/END, SYNC_DATA_START)
